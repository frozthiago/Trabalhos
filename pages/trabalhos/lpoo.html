Exercícios de Padrões de Projeto - Resolvidos

Esta página demonstra a aplicação de diversos Padrões de Projeto em cenários práticos, conforme exercícios propostos em minha formação acadêmica. Cada seção aborda um padrão específico, detalhando sua aplicação e os benefícios que ele oferece.

Parte 1: Implementação de Padrões de Projeto

1. Singleton – Configurações Globais do Sistema

Problema: Uma empresa de software precisa garantir que as configurações de idioma e fuso horário sejam únicas e acessíveis globalmente.

Solução: Criar uma classe ConfiguracaoSistema que utilize o padrão Singleton para garantir que só exista uma instância no sistema.

  • O sistema deve permitir alterar e exibir o idioma e o fuso horário.
  • Demonstrar que ao alterar as configurações em um local, a mudança reflete em todo o sistema.
Resolução: O padrão Singleton é ideal para este cenário, pois garante que apenas uma instância da classe de configuração seja criada e que ela seja globalmente acessível. Isso evita inconsistências e duplicação de dados de configuração.

2. Factory Method – Criação de Documentos

Problema: Uma ferramenta de escritório digital deve permitir que o usuário crie documentos em diferentes formatos (PDF, Word, Planilha) sem precisar saber a lógica de construção.

Solução: Implementar uma fábrica usando o padrão Factory Method para criar objetos de cada tipo, cada um com um método abrir().

  • O programa principal deve receber do usuário o tipo de documento e abri-lo.
Resolução: O Factory Method permite que a lógica de criação de objetos seja encapsulada em subclasses, tornando o sistema mais flexível e extensível. O cliente (programa principal) interage apenas com a interface da fábrica, sem se preocupar com os detalhes de implementação dos tipos de documentos.

3. Builder – Montagem de Computadores

Problema: Uma loja de informática deseja permitir que clientes montem computadores personalizados.

Solução: Criar a classe Computador com atributos como processador, memoriaRAM, armazenamento e placaVideo. Usar o padrão Builder para criar computadores de forma flexível.

  • Exemplo: um computador básico para escritório e outro “gamer” com alto desempenho.
Resolução: O padrão Builder é perfeito para construir objetos complexos passo a passo. Ele separa a construção de um objeto de sua representação, permitindo que o mesmo processo de construção crie diferentes representações do objeto. Isso é ideal para a montagem de computadores com diversas configurações.

4. Prototype – Clonagem de Figuras

Problema: Um software de design precisa criar rapidamente cópias de figuras gráficas mantendo suas propriedades.

Solução: Criar a classe Figura com atributos como cor e tamanho e implementar o método clone().

  • Criar um objeto original e gerar um clone mudando apenas a cor.
  • Mostrar que o clone mantém as demais características.
Resolução: O padrão Prototype permite criar novos objetos copiando um objeto existente (o protótipo). Isso é mais eficiente do que criar um novo objeto do zero, especialmente quando o processo de criação é complexo ou caro. A clonagem garante que as propriedades originais sejam mantidas, facilitando modificações pontuais.

5. Adapter – Compatibilidade de Formatos de Áudio

Problema: Um sistema de música possui um player que aceita apenas arquivos MP3, mas agora precisa reproduzir arquivos WAV.

Solução: Implementar uma interface ReprodutorAudio e um ReprodutorMP3 compatível. Usar o padrão Adapter para permitir que um ReprodutorWAV funcione no mesmo sistema sem modificar o código original.

Resolução: O padrão Adapter atua como uma ponte entre duas interfaces incompatíveis. Ele permite que objetos com interfaces diferentes colaborem, sem a necessidade de modificar o código existente. Neste caso, o Adapter "traduz" a interface do ReprodutorWAV para que seja compatível com o sistema que espera um ReprodutorMP3.

6. Decorator – Personalização de Café

Problema: Uma cafeteria quer permitir que clientes montem cafés com ingredientes extras pagando valores adicionais.

Solução: Criar uma interface Cafe com o método custo(). Classe base: CafeSimples. Decoradores: Leite, Chocolate, Chantilly. Usar o padrão Decorator para adicionar funcionalidades sem mudar o código do café base.

Resolução: O padrão Decorator permite adicionar novas funcionalidades a um objeto dinamicamente, sem alterar sua estrutura. Ele é uma alternativa flexível à herança para estender funcionalidades. Cada "decorador" (Leite, Chocolate, Chantilly) envolve o objeto de café base, adicionando seu próprio custo e descrição.

7. Observer – Sistema de Notificação de Clima

Problema: Um centro meteorológico precisa notificar diferentes canais quando o clima mudar.

Solução: Implementar o padrão Observer onde a classe EstacaoClimatica notifica DisplayTV e AplicativoClima sobre mudanças de clima.

  • Quando a estação mudar o estado, todos os observadores devem ser informados automaticamente.
Resolução: O padrão Observer define uma dependência um-para-muitos entre objetos, de modo que quando um objeto muda de estado, todos os seus dependentes são notificados e atualizados automaticamente. A Estação Climática é o "sujeito" e os Displays de TV e Aplicativos de Clima são os "observadores".

8. Strategy – Cálculo de Frete

Problema: Um e-commerce deseja calcular frete com diferentes estratégias: Sedex, PAC e RetiradaLocal.

Solução: Implementar a interface EstrategiaFrete com o método calcular(double peso). O sistema deve permitir trocar a estratégia em tempo de execução sem alterar o código principal.

Resolução: O padrão Strategy permite definir uma família de algoritmos, encapsular cada um deles e torná-los intercambiáveis. A estratégia de frete pode ser alterada dinamicamente, sem modificar o código que a utiliza. Isso promove a flexibilidade e a manutenção do código.

9. Command – Controle Remoto

Problema: Uma empresa de automação residencial quer criar um controle remoto universal que execute comandos em dispositivos diferentes.

Solução: Implementar a interface Comando e criar comandos como LigarTV, DesligarTV e AumentarVolume. O padrão Command deve encapsular a ação e permitir que o controle remoto execute qualquer comando atribuído.

Resolução: O padrão Command encapsula uma solicitação como um objeto, permitindo parametrizar clientes com diferentes solicitações, enfileirar ou registrar solicitações e suportar operações desfazíveis. O controle remoto não precisa saber os detalhes da operação, apenas invocar o comando.

10. Template Method – Relatórios

Problema: Uma empresa gera relatórios financeiros e de vendas com um fluxo comum: Coletar dados, Processar, Formatar, Exibir.

Solução: Criar uma classe abstrata Relatorio usando o padrão Template Method e subclasses que implementem detalhes específicos para RelatorioFinanceiro e RelatorioVendas.

Resolução: O padrão Template Method define o esqueleto de um algoritmo em uma operação, adiando alguns passos para subclasses. Ele permite que as subclasses redefinam certos passos de um algoritmo sem mudar a estrutura do algoritmo. Isso é ideal para relatórios com etapas comuns, mas detalhes de processamento diferentes.

Parte 2: Questões de Múltipla Escolha

1. Um software corporativo precisa controlar a quantidade de licenças ativas no sistema, garantindo que todas as partes do programa consultem a mesma instância responsável por esse controle. Qual padrão deve ser usado?

  • A) Builder
  • B) Adapter
  • C) Singleton
  • D) Strategy
  • E) Prototype
Resposta Correta: C) Singleton. O Singleton garante que uma classe tenha apenas uma instância e fornece um ponto de acesso global a ela, ideal para controle de licenças.

2. Um banco digital precisa criar contas de tipos diferentes (corrente, poupança, salário) dependendo da solicitação do cliente, sem que o código principal precise conhecer a lógica de criação de cada tipo. Qual padrão resolve isso?

  • A) Factory Method
  • B) Singleton
  • C) Template Method
  • D) Decorator
  • E) Observer
Resposta Correta: A) Factory Method. O Factory Method define uma interface para criar um objeto, mas permite que as subclasses decidam qual classe instanciar, delegando a lógica de criação.

3. Uma montadora quer permitir que clientes configurem um carro escolhendo motor, cor, tipo de banco e acessórios. Cada configuração deve gerar um objeto final pronto para uso. Qual padrão de projeto é o mais indicado?

  • A) Adapter
  • B) Builder
  • C) Prototype
  • D) Command
  • E) Strategy
Resposta Correta: B) Builder. O Builder é usado para construir objetos complexos passo a passo, permitindo diferentes configurações do mesmo objeto.

4. Em um jogo de RPG, o jogador pode criar vários personagens iguais a um modelo existente e apenas modificar alguns atributos, como cor da roupa ou arma utilizada. Qual padrão é mais apropriado?

  • A) Prototype
  • B) Builder
  • C) Strategy
  • D) Command
  • E) Singleton
Resposta Correta: A) Prototype. O Prototype permite criar novos objetos copiando um objeto existente, o que é eficiente para criar muitas instâncias semelhantes com pequenas variações.

5. Um sistema de e-commerce utiliza um módulo de pagamento que só aceita dados em formato XML, mas a nova API de pagamentos fornece dados em JSON. Qual padrão é adequado para compatibilizar os dois formatos sem modificar o código original?

  • A) Adapter
  • B) Template Method
  • C) Observer
  • D) Builder
  • E) Command
Resposta Correta: A) Adapter. O Adapter permite que interfaces incompatíveis trabalhem juntas, convertendo a interface de uma classe para outra que o cliente espera.

6. Um aplicativo de mensagens permite adicionar funcionalidades extras às notificações, como sons diferentes, ícones personalizados e vibração especial, sem modificar a classe de notificação original. Qual padrão é usado?

  • A) Decorator
  • B) Strategy
  • C) Prototype
  • D) Observer
  • E) Command
Resposta Correta: A) Decorator. O Decorator permite adicionar responsabilidades a objetos individualmente e dinamicamente, sem alterar sua estrutura.

7. Um supermercado quer que vários sistemas (caixas, controle de estoque, compras online) sejam atualizados automaticamente sempre que o estoque de um produto mudar. Qual padrão se aplica?

  • A) Builder
  • B) Singleton
  • C) Observer
  • D) Command
  • E) Template Method
Resposta Correta: C) Observer. O Observer define uma dependência um-para-muitos, onde múltiplos objetos são notificados automaticamente quando o estado de um objeto muda.

8. Um sistema contábil precisa calcular impostos de acordo com o regime da empresa (Simples Nacional, Lucro Presumido ou Lucro Real). A fórmula de cálculo muda, mas o processo geral é o mesmo. Qual padrão é o mais indicado?

  • A) Factory Method
  • B) Strategy
  • C) Observer
  • D) Singleton
  • E) Prototype
Resposta Correta: B) Strategy. O Strategy permite que diferentes algoritmos (estratégias de cálculo de imposto) sejam encapsulados e trocados em tempo de execução.

9. Um editor de texto possui ações como "copiar", "colar" e "desfazer", que devem ser encapsuladas para que possam ser executadas e revertidas quando necessário. Qual padrão representa essa abordagem?

  • A) Strategy
  • B) Command
  • C) Template Method
  • D) Observer
  • E) Builder
Resposta Correta: B) Command. O Command encapsula uma solicitação como um objeto, permitindo que as operações sejam enfileiradas, registradas e desfeitas.

10. Um sistema de leitura de arquivos segue sempre o mesmo fluxo: abrir o arquivo, processar os dados, gerar o relatório e fechar o arquivo. O processamento é diferente para arquivos CSV e XML, mas as etapas principais são fixas. Qual padrão representa essa solução?

  • A) Template Method
  • B) Adapter
  • C) Factory Method
  • D) Decorator
  • E) Singleton
Resposta Correta: A) Template Method. O Template Method define o esqueleto de um algoritmo em uma superclasse, permitindo que as subclasses implementem ou redefinam certas etapas sem alterar a estrutura geral.
